# Quantified Acceptance

将模糊的验收陈述转化为具体的、可测量的标准。

## When to Use

- 撰写或审核 `task-breakdown.md` 的验收标准时
- 审核 `acceptance-plan.md` 的验收条款时
- 将 user stories 转化为可测试的任务时
- 任何时候发现验收标准是软陈述而非硬指标

## NOT When to Use

- 验收标准已经明确量化（如"响应时间 ≤ 200ms"）
- 项目处于探索阶段，指标尚未稳定

## Input

| 来源 | 内容 |
|------|------|
| task-breakdown.md | 任务拆分和验收标准 |
| user-stories.md | 用户故事和完成标准 |
| acceptance-plan.md | 验收计划 |

## Output Schema

| 文件 | 类型 | 说明 |
|------|------|------|
| `wiki/06-quality/quantified-checklist.md` | 量化检查清单 | 可直接用于测试的验收标准 |

## Minimum Viable Output

- 至少 5 条验收标准的量化版本
- 每条标准包含：指标名称、目标值、可接受上限/下限、测试方法

## Complete Output

- 全量验收标准的量化版本
- 每条标准的测试方法（如何验证）
- 优先级标注（P0/P1/P2）
- 覆盖所有 US 的验收矩阵

## Dependencies

| 类型 | 说明 |
|------|------|
| 前置 | `delivery-planning`（task breakdown 已输出）|
| 前置 | `requirements-spec`（user stories 已输出）|
| 嵌入 | `delivery-planning` Procedure |
| 依赖读取 | task-breakdown.md、user-stories.md |

## Procedure

### Phase 1: 收集所有验收陈述

从 task-breakdown.md 和 acceptance-plan.md 中提取所有验收标准，标注每一条的原文：

```markdown
## 验收陈述收集

| ID | 来源 | 原文陈述 |
|----|------|---------|
| AC-001 | task-breakdown.md:T1 | 开发环境可运行，能执行单元测试 |
| AC-002 | task-breakdown.md:T2 | 核心资源管理可正常使用 |
| AC-003 | acceptance-plan.md | 核心提交操作响应时间 ≤ 3 秒 |
```

### Phase 2: 识别不可测量的陈述

以下模式是**不可测量**的，需要量化：

```
❌ "可运行" → 什么条件算"可运行"？
❌ "功能完整" → 哪些功能？完整度如何衡量？
❌ "用户体验良好" → 什么是"良好"？
❌ "操作便捷" → 几步算便捷？
❌ "正常处理" → 什么算"正常"？
❌ "及时提醒" → 多及时算及时？
```

### Phase 3: 量化转换模板

```markdown
## 量化转换模板

原始陈述：[原文]

量化版本：

| 字段 | 值 |
|------|-----|
| 指标名称 | [具体可度量的指标] |
| 目标值 | [期望达到的值] |
| 可接受范围 | [min, max] 或 [≥ X, ≤ Y] |
| 测量方法 | [如何在测试中验证] |
| 测量工具 | [用什么工具/方法测量] |
| 优先级 | P0 / P1 / P2 |
```

### Phase 4: 量化示例

```markdown
## 量化示例

### 不可测量 → 可测量

**原始**：开发环境可运行

**量化后**：

| 字段 | 值 |
|------|-----|
| 指标名称 | 编译成功率 |
| 目标值 | 100% |
| 可接受范围 | ≥ 99% |
| 测量方法 | `flutter build apk --debug` 退出码为 0 |
| 测量工具 | CI 脚本 |
| 优先级 | P0 |

---

**原始**：核心资源管理可正常使用

**量化后**：

| 字段 | 值 |
|------|-----|
| 指标名称 | 核心资源 CRUD 操作成功率 |
| 目标值 | ≥ 99% |
| 可接受范围 | ≥ 95% |
| 测量方法 | 自动化测试覆盖 创建/读取/更新/删除 四个场景 |
| 测量工具 | flutter test |
| 优先级 | P0 |

---

**原始**：应用冷启动时间 ≤ 5 秒

**量化后**：

| 字段 | 值 |
|------|-----|
| 指标名称 | 应用冷启动时间（P50） |
| 目标值 | ≤ 3 秒 |
| 可接受范围 | [0, 5 秒] |
| 测量方法 | 测量应用从 launch 到首帧渲染的时间 |
| 测量工具 | Flutter benchmark |
| 优先级 | P1 |
```

### Phase 5: 生成量化检查清单

```markdown
## 量化检查清单

### P0 — 必须满足（阻塞发布）

| ID | 指标 | 目标 | 测试方法 | 状态 |
|----|------|------|---------|------|
| QA-P0-001 | 构建成功率 | 100% | 项目构建命令 | ⬜ |
| QA-P0-002 | 单元测试通过率 | 100% | 项目测试命令 | ⬜ |
| QA-P0-003 | 核心操作成功率 | ≥ 99% | 自动化测试 | ⬜ |

### P1 — 重要（不阻塞但需跟进）

| ID | 指标 | 目标 | 测试方法 | 状态 |
|----|------|------|---------|------|
| QA-P1-001 | 冷启动时间 | ≤ 3s | benchmark | ⬜ |
| QA-P1-002 | 核心操作响应时间 | ≤ 3s | 自动化计时 | ⬜ |

### P2 — 优化项（可选）

| ID | 指标 | 目标 | 测试方法 | 状态 |
|----|------|------|---------|------|
| QA-P2-001 | 内存占用峰值 | ≤ 200MB | 性能监控 | ⬜ |
```

### Phase 6: 与 US 的映射

验证每个 US 的完成标准都被量化检查清单覆盖：

```markdown
## US 覆盖矩阵

| US-ID | US 描述 | 对应 QA 项 | 覆盖率 |
|-------|---------|-----------|-------|
| US-001 | 创建资源 | QA-P0-003 | ✅ 100% |
| US-002 | 更新资源 | QA-P0-003 | ✅ 100% |
| US-003 | 批量导入 | - | ❌ 未覆盖 |
```

未覆盖的 US 需要补充对应的量化标准。

### US 覆盖率计算公式

```
US 覆盖率 = (已覆盖 US 数 / 总 US 数) × 100%

已覆盖 US = 该 US 至少有一条量化验收标准（QA 项）
```

**判定标准**：
- 覆盖率 ≥ 80%：✅ 可接受
- 覆盖率 < 80%：⚠️ 需补充缺失的量化标准
- 覆盖率 < 50%：❌ 量化工作严重不足，需重新审视验收标准设计

## 量化阈值参考表

| 类型 | 常见指标 | 典型目标值 |
|------|---------|-----------|
| 性能 | 响应时间 | ≤ 200ms (P50), ≤ 1s (P99) |
| 性能 | 启动时间 | ≤ 3s (冷), ≤ 1s (热) |
| 可用性 | 成功率 | ≥ 99% |
| 可用性 | 错误率 | ≤ 1% |
| 资源 | 内存占用 | ≤ 200MB |
| 资源 | 包大小 | ≤ 50MB |
| 安全 | 认证失败锁定 | 5 次后锁定 30 分钟 |
| 安全 | 密码强度 | ≥ 8 字符，含数字和特殊字符 |
