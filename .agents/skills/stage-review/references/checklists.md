# Stage Review Checklists

## Bootstrap (4 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R1.1 | HOME.md 含一句话定义 | 存在且 ≤50字 |
| R1.2 | 状态三要素齐全 | 当前阶段 + 当前目标 + 下一步 均非空 |
| R1.3 | 导航链接覆盖 | wiki/00 + wiki/01 + wiki/03 至少含 2 个 |
| R1.4 | 未堆长文 | 无超过 5 行的段落描述详细设计内容 |

## Incubation (4 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R2.1 | 用户画像 ≥2 个 | 每个有名称+特征描述 |
| R2.2 | 场景 ≥2 个 | 每个含触发条件+预期行为 |
| R2.3 | 候选方向 ≥3 个 | 每个有简要说明(非仅标题) |
| R2.4 | 未伪装定稿 | 候选方向不含"确定""最终""选定"等定稿用语 |

## Overview (5 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R3.1 | vision.md: 问题陈述 | 非空，非泛化("存在效率问题"→不通过，需具体) |
| R3.2 | vision.md: 目标用户 | 非空，非"所有人"等无意义表述 |
| R3.3 | vision.md: 成功标准 ≥1 条 | 含数字或明确判定条件("≤3分钟""≥80%") |
| R3.4 | scope.md: 范围内/外 | 两个清单均有 ≥3 条内容 |
| R3.5 | scope.md: 非目标 ≥3 条 | 每条具体，非"不需要的功能"这种空泛表述 |

## Requirements (4 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R4.1 | user-stories.md: ≥3 条 | 格式为"作为...我希望...以便..." |
| R4.2 | MVP/后续划分 | 每条标注 MVP 或 后续迭代 |
| R4.3 | constraints.md: ≥2 条 | 每条具体有可验证的影响 |
| R4.4 | 无可验证性空洞 | 不存在"系统应该好用""体验要好"等不可验证表述 |

## Architecture (5 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R5.1 | 模块 ≥3 个 | 每个有名称+职责描述 |
| R5.2 | 关键链路 ≥1 条 | 含 输入→处理→输出 三段结构 |
| R5.3 | 🔴/🟡 分级已标注 | 至少 1 条 🔴 不可逆决策 |
| R5.4 | 候选技术栈已列 | ≥2 个候选方案(保留选择，不强行定稿) |
| R5.5 | 技术假设 ≥2 条 | 存在 `[AUTO-ASSUMPTION: ...]` 标注 |

## Delivery (5 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| R6.1 | 任务 ≥5 条 | 每条有 ID + 名称 + 依赖 |
| R6.2 | 里程碑 ≥2 个 | 每个有阶段目标 |
| R6.3 | 风险简报 ≥3 条 | 每条指向具体 `[AUTO-ASSUMPTION]` 假设 |
| R6.4 | 依赖闭合 | 无孤立任务(0依赖+0被依赖)，除非是 T1 |
| R6.5 | MVP 范围标注 | 哪些任务是 MVP 必需已标明 |

## Cross-stage Coherence (4 项)

| ID | 检查项 | 判定标准 |
|----|--------|---------|
| C1 | 架构模块 → 需求映射 | 每个架构模块至少服务 1 条用户故事 |
| C2 | 任务 → 架构映射 | 每个架构模块至少对应 1 条交付任务 |
| C3 | 假设一致性 | 无前后矛盾(如阶段2假设Web-only，阶段5讨论移动端) |
| C4 | 无产出断裂 | 前阶段结论在后续阶段被引用(非独立重写) |
